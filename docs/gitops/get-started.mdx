title: Get Started

This guide walks you through setting up a GitOps workflow with Bytebase. By the end, you'll have automated SQL review on pull/merge requests and automated database deployment on merge.

## Prerequisites

- A running Bytebase instance ([Cloud](https://hub.bytebase.com/) or [self-hosted](/get-started/self-host/deploy-with-docker))
- A GitHub or GitLab repository for your database migrations
- At least one database instance registered in Bytebase

## Step 1 — Authenticate CI/CD with Bytebase

Your CI/CD pipeline needs to call the Bytebase API. Choose one of the following methods:

<Tabs>
  <Tab title="Workload Identity (recommended)">
    Workload Identity uses short-lived OIDC tokens — no secrets to store or rotate.

    1. In Bytebase, go to **IAM & Admin > Workload Identity** and create a new identity for your repository.
    2. Assign it the **GitOps Service Agent** role in your project.

    See the setup guide for your platform:
    - [GitHub Actions Workload Identity](/administration/workload-identity/github-actions)
    - [GitLab CI Workload Identity](/administration/workload-identity/gitlab-ci)
  </Tab>
  <Tab title="Service Account">
    Service accounts use long-lived API keys stored as CI/CD secrets.

    1. In Bytebase, go to **IAM & Admin > Users & Groups** and click **+ Add User**.
    2. Create a service account (e.g., `api-gitops`) and assign it the **Workspace Member** and **GitOps Service Agent** roles.
    3. Copy the service key and add it as a secret in your CI/CD platform:

    <Tabs>
      <Tab title="GitHub">
        Go to **Settings > Secrets and Variables > Actions** and add `BYTEBASE_SERVICE_ACCOUNT_SECRET`.
      </Tab>
      <Tab title="GitLab">
        Go to **Settings > CI/CD > Variables** and add `BYTEBASE_SERVICE_ACCOUNT_SECRET` (masked).
      </Tab>
    </Tabs>

    See [Service Account](/administration/service-account) for more details.
  </Tab>
</Tabs>

## Step 2 — Configure Runners

CI/CD workflows run on **runners** that need network access to your Bytebase instance.

<Tabs>
  <Tab title="GitHub">
    | Bytebase Setup | Runner Type |
    |---|---|
    | Bytebase Cloud or public URL | GitHub-hosted runners (default, no setup needed) |
    | Private network | [Self-hosted runners](https://docs.github.com/en/actions/hosting-your-own-runners/managing-self-hosted-runners/adding-self-hosted-runners) in the same network |
  </Tab>
  <Tab title="GitLab">
    | Bytebase Setup | Runner Type |
    |---|---|
    | Bytebase Cloud or public URL | GitLab shared runners (default on GitLab.com, no setup needed) |
    | Private network | [Self-managed runners](https://docs.gitlab.com/runner/install/) in the same network |

    For self-hosted GitLab with no internet access, you'll need to pre-load the `bytebase/bytebase-action` Docker image on your runner. See the [GitLab tutorial](/tutorials/gitops-gitlab-workflow#self-hosted-gitlab-considerations) for details.
  </Tab>
</Tabs>

## Step 3 — Generate Workflow Configuration Files

Go to the **GitOps** page in your Bytebase project to generate the CI and CD configuration files for your repository.

{/* TODO: Add screenshot of GitOps landing page */}

The tool generates two workflow files:

- **SQL Review (CI)** — Triggers on pull/merge requests to review SQL changes and post results.
- **Release (CD)** — Triggers on merge to `main` to create a Bytebase release and roll out changes.

Download the generated files and place them in your repository:

<Tabs>
  <Tab title="GitHub">
    - `.github/workflows/sql-review.yml`
    - `.github/workflows/release.yml`
  </Tab>
  <Tab title="GitLab">
    - `.gitlab-ci.yml` (includes both review and rollout stages)
  </Tab>
</Tabs>

Commit and push to your `main` branch.

<Accordion title="Advanced: manual configuration">
If you prefer to configure the workflow files manually, see the example repositories:
- GitHub: [bytebase/example-gitops-github-flow](https://github.com/bytebase/example-gitops-github-flow)
- GitLab: [bytebase-sample/gitops-example](https://gitlab.com/bytebase-sample/gitops-example)

Key environment variables to set:
- **`BYTEBASE_URL`** — Your Bytebase instance URL
- **`BYTEBASE_SERVICE_ACCOUNT`** — Service account email
- **`BYTEBASE_PROJECT`** — Project resource name (e.g., `projects/my-project`)
- **`BYTEBASE_TARGETS`** — Comma-separated database resource names
- **`FILE_PATTERN`** — Migration file glob pattern (e.g., `migrations/*.sql`)
</Accordion>

## Step 4 — Try Your First GitOps Migration

### Create a migration file

1. Create a new branch:

   ```bash
   git checkout -b add-employees-table
   ```

2. Add a migration file under your migrations directory (e.g., `migrations/`):

   **`migrations/20260227000000_create_employees_ddl.sql`**
   ```sql
   CREATE TABLE employees (
       id SERIAL PRIMARY KEY,
       name TEXT NOT NULL,
       email TEXT NOT NULL
   );
   ```

   <Accordion title="Migration file naming convention">
   Migration files must start with digits (the version number), e.g.:
   - Timestamp-based: `20260227000000_create_employees_ddl.sql`
   - Semver-based: `1.0.0_create_employees_ddl.sql`

   Files may end with `_ddl` or `_dml` to indicate the change type. If omitted, DDL is assumed.
   </Accordion>

3. Commit and push:

   ```bash
   git add . && git commit -m "Add employees table" && git push -u origin add-employees-table
   ```

### Review — SQL review triggers automatically

4. Open a pull request (GitHub) or merge request (GitLab). The SQL review workflow triggers and posts results.

   <Tabs>
     <Tab title="GitHub">
       ![GitHub PR SQL Review](/content/docs/gitops/sql-review-ci/github-pr-sql-review.webp)
     </Tab>
     <Tab title="GitLab">
       ![GitLab MR SQL Review](/content/docs/gitops/sql-review-ci/gitlab-pr-sql-review.webp)
     </Tab>
   </Tabs>

   If there are warnings or errors, update your SQL file and push again — the review re-runs automatically.

### Deploy — Release triggers on merge

5. Once SQL review passes, merge. The release workflow triggers, creates a Bytebase release, and rolls out to your target databases.

6. Check the CI/CD pipeline to see the release created and applied to each database.

   <Tabs>
     <Tab title="GitHub">
       Go to the **Actions** tab to see the workflow run and job summary.
     </Tab>
     <Tab title="GitLab">
       Go to **CI/CD > Pipelines** to see the pipeline stages and logs.
     </Tab>
   </Tabs>

You now have a working GitOps pipeline — SQL review on every change, automated deployment on merge.
