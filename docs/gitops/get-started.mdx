title: Get Started
---

Set up GitOps with Bytebase in 4 steps. By the end, you'll have SQL review on every pull/merge request and automated database deployment on merge.

## Prerequisites

- A running Bytebase instance ([Cloud](https://hub.bytebase.com/) or [self-hosted](/get-started/self-host/deploy-with-docker))
- A GitHub or GitLab repository
- At least one database registered in a Bytebase project

## Step 1 — Set Up Workload Identity

[Workload Identity](/administration/workload-identity/overview) lets your CI/CD pipeline authenticate with Bytebase using short-lived OIDC tokens — no secrets to manage.

<Tabs>
  <Tab title="GitHub">
    1. In Bytebase, go to **IAM & Admin > Users & Groups**, click **Add User**, and select **Workload Identity**.
    2. Choose **GitHub Actions** as the platform and fill in your repository details.
    3. Assign the **GitOps Service Agent** role.

    See [GitHub Actions Workload Identity](/administration/workload-identity/github-actions) for the full setup.
  </Tab>
  <Tab title="GitLab">
    1. In Bytebase, go to **IAM & Admin > Users & Groups**, click **Add User**, and select **Workload Identity**.
    2. Choose **GitLab CI** as the platform and fill in your group/project details.
    3. Assign the **GitOps Service Agent** role.

    See [GitLab CI Workload Identity](/administration/workload-identity/gitlab-ci) for the full setup.
  </Tab>
</Tabs>

<Accordion title="Alternative: Service Account">
You can also use a [Service Account](/administration/service-account) with a long-lived API key. Create one in **IAM & Admin > Users & Groups** with the **GitOps Service Agent** role, then store the key as a CI/CD secret. Workload Identity is recommended for production use.
</Accordion>

## Step 2 — Configure Runners

Runners execute your CI/CD workflows and need network access to Bytebase.

- **Bytebase Cloud or public URL** — Use the default cloud-hosted runners. No setup needed.
- **Bytebase in a private network** — Use a self-hosted runner in the same network.

<Tabs>
  <Tab title="GitHub">
    GitHub-hosted runners work out of the box for public Bytebase instances. For private networks, see [GitHub self-hosted runners](https://docs.github.com/en/actions/hosting-your-own-runners/managing-self-hosted-runners/adding-self-hosted-runners).
  </Tab>
  <Tab title="GitLab">
    GitLab shared runners (default on GitLab.com) work out of the box for public Bytebase instances. For private networks, see [GitLab self-managed runners](https://docs.gitlab.com/runner/install/).
  </Tab>
</Tabs>

## Step 3 — Generate Workflow Files

In Bytebase, go to your project's **GitOps** page to generate the CI/CD configuration files.

{/* TODO: Add screenshot of GitOps landing page */}

The page generates two workflow configurations:

- **SQL Review (CI)** — Reviews SQL changes on every pull/merge request.
- **Release (CD)** — Creates a Bytebase release and deploys on merge to `main`.

Download the files, place them in your repository, commit, and push to `main`.

<Tabs>
  <Tab title="GitHub">
    Place the generated files at:
    - `.github/workflows/sql-review.yml`
    - `.github/workflows/release.yml`
  </Tab>
  <Tab title="GitLab">
    Place the generated file at:
    - `.gitlab-ci.yml`
  </Tab>
</Tabs>

<Accordion title="Manual configuration">
If you prefer to write the workflow files manually, see the example repositories:
- [GitHub example](https://github.com/bytebase/example-gitops-github-flow)
- [GitLab example](https://gitlab.com/bytebase-sample/gitops-example)
</Accordion>

## Step 4 — Run Your First Migration

### Write a migration

1. Create a branch and add a migration file:

   ```bash
   git checkout -b add-employees-table
   ```

   **`migrations/20260227000000_create_employees_ddl.sql`**

   ```sql
   CREATE TABLE employees (
       id SERIAL PRIMARY KEY,
       name TEXT NOT NULL,
       email TEXT NOT NULL
   );
   ```

   <Accordion title="File naming convention">
   Files must start with a version number:
   - `20260227000000_create_employees_ddl.sql` (timestamp)
   - `1.0.0_create_employees_ddl.sql` (semver)

   Suffix `_ddl` or `_dml` indicates the change type. Default is DDL.
   </Accordion>

2. Push and open a pull request (GitHub) or merge request (GitLab):

   ```bash
   git add . && git commit -m "Add employees table" && git push -u origin add-employees-table
   ```

### SQL review runs automatically

3. The SQL review workflow triggers and posts results to your pull/merge request.

   <Tabs>
     <Tab title="GitHub">
       ![GitHub PR SQL Review](/content/docs/gitops/sql-review-ci/github-pr-sql-review.webp)
     </Tab>
     <Tab title="GitLab">
       ![GitLab MR SQL Review](/content/docs/gitops/sql-review-ci/gitlab-pr-sql-review.webp)
     </Tab>
   </Tabs>

   Fix any issues, push again, and the review re-runs.

### Merge triggers deployment

4. Merge the pull/merge request. The release workflow creates a Bytebase release and rolls out to your databases.

   <Tabs>
     <Tab title="GitHub">
       Go to **Actions** to see the workflow run and deployment summary.
     </Tab>
     <Tab title="GitLab">
       Go to **CI/CD > Pipelines** to see the pipeline stages and deployment logs.
     </Tab>
   </Tabs>

That's it — SQL review on every change, automated deployment on merge.
